# TRE2026 Three-Day Calendar Design

## Goal

將 tab2 從原本偏「候補活動衝突分析」的列表頁，改成適合手機觀看的 `7/3`、`7/4`、`7/5` 三日月曆式行程檢視。主要用途是讓使用者快速看到自己已購活動落在每天哪個時段，以及哪些時段會和主辦方固定權益衝突。

## Confirmed Scope

- 顯示型態：三日週檢視風格
- 裝置重點：手機瀏覽
- 呈現方式：保留月曆感，但允許橫向滑動
- 日期範圍：只顯示 `2026/07/03`、`2026/07/04`、`2026/07/05`
- 資料來源：
  - 使用者已購活動：瀏覽器 local storage
  - 主辦方固定權益：既有 app data 規則與事件資料
- 第一版重點：看行程位置與衝突，不做複雜編輯

## Recommended Approach

採用「三欄日曆 + 橫向滑動 + 縱向時間軸」：

- 每一天是一個固定寬度的日曆欄
- 手機上三欄可左右滑動
- 每欄有相同的時間軸
- 已購活動以絕對定位事件卡呈現到對應開始與結束時間
- 主辦方固定權益以背景區塊或特殊卡片標示

這個方案同時保留你要的「週檢視月曆感」與手機可讀性，比硬塞三欄到窄螢幕更穩。

## UI Structure

### Header

- 標題：`三日行程月曆`
- 說明：提示此頁只顯示 `7/3`、`7/4`、`7/5`
- 摘要徽章：
  - 已購活動數
  - 主辦方權益時段數
  - 衝突數

### Calendar Grid

- 外層為可橫向滑動容器
- 內層為三欄：
  - `7/3 Fri`
  - `7/4 Sat`
  - `7/5 Sun`
- 每欄結構：
  - 日期標頭
  - 時間軸
  - 事件圖層

### Event Blocks

事件卡至少顯示：

- 活動名稱
- 人物名稱
- 時段
- 類型標記

類型分為：

- 使用者已購活動
- 金卡固定權益
- 白銀卡固定權益

### Conflict Treatment

若已購活動和金卡或白銀卡權益時段重疊：

- 活動卡加上紅色描邊或紅底標記
- 權益區塊也顯示警示樣式
- 事件卡內附簡短提示，例如：
  - `撞到金卡`
  - `撞到白銀卡`
  - `撞到 2 個時段`

## Data Flow

1. 讀取 tab1/tab3 共用的 `UserScheduleRecord`
2. 將 `purchasedEntries` 依日期分到 `7/3`、`7/4`、`7/5`
3. 從既有 `AppData` 中取出主辦方固定權益事件
4. 將兩種事件轉成共用 calendar block view model
5. 套用時間轉像素定位
6. 顯示在對應日欄中

## Time Window Assumptions

第一版建議固定時間軸範圍：

- `10:00` 到 `22:00`

原因：

- 足以涵蓋目前活動常見時間
- 高度可控，手機上不會過長
- 若事件落在範圍外，再擴充即可

若某事件早於或晚於這個範圍：

- 自動擴充上下界，避免事件被裁掉

## Component Plan

建議維持 tab2 功能集中，不先過度拆分：

- `ConflictAnalysisTab.tsx`
  - 改為三日月曆頁容器
- `TimelineView.tsx`
  - 改造成 calendar grid renderer，或更名但先不強制
- 新增 view-model helpers
  - 產生日欄資料
  - 計算 top / height
  - 標示衝突狀態

## Testing Strategy

至少覆蓋：

- 只顯示 `7/3`、`7/4`、`7/5`
- 已購活動會落在正確日期欄
- 金卡 / 白銀卡權益出現在正確日期欄
- 發生重疊時顯示衝突標示
- 無已購活動時仍顯示空月曆框架

## Recommendation

這次先把 tab2 專注成「我的三日行程月曆」，不要再承擔候補活動切換分析。候補活動衝突若之後還要保留，應另做一個獨立頁面或次要模式，否則 tab2 會同時服務兩種任務，操作與資料模型都會開始互相牽制。
